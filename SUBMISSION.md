# AI Container Orchestrator

> **자연어 한 줄로 베어메탈 서버의 컨테이너를 제어하는 멀티노드 오케스트레이션 플랫폼**

**트랙**: B — 업무 자동화/혁신 툴
**팀**: 1인 개발 (Claude Code AI 활용)

---

## 프로젝트 개요

### 해결하려는 문제

파라메타는 블록체인 인프라를 운영하기 위해 다수의 베어메탈 서버를 관리합니다. 현재 이 과정에서 두 가지 극단 사이에 놓여 있습니다:

| 도구 | 문제점 |
|------|--------|
| **Kubernetes** | 학습 곡선이 가파르고, 베어메탈 환경에서 설치·운영이 과도하게 복잡. 소규모 클러스터에는 오버엔지니어링 |
| **Docker Compose** | 단일 호스트에 한정. 멀티노드 배포, 자동 복구, 스케줄링 불가 |
| **수동 SSH 관리** | 서버마다 직접 접속하여 CLI로 조작. 반복적이고 오류 발생이 잦음. 비개발자 접근 불가 |

### 솔루션

**AI Container Orchestrator**는 Kubernetes의 핵심 기능(멀티노드 스케줄링, 자동 복구, 서비스 디스커버리)을 **단일 바이너리(~17MB)**로 제공하면서, **한국어/영어 자연어 명령**으로 모든 인프라 작업을 수행할 수 있게 합니다.

```
"nginx를 3개 배포해줘" → 자동으로 최적 노드 선택 → 컨테이너 생성 → 라우팅 설정 → 모니터링 시작
```

비개발자도 대시보드의 AI 채팅창에서 인프라를 제어할 수 있어, DevOps 지식 없이도 운영이 가능합니다.

## 핵심 기능

### 1. 자연어 컨테이너 제어 (한국어/영어)
한국어와 영어 자연어 명령을 패턴 매칭 엔진으로 파싱하여 컨테이너 작업으로 변환합니다. "redis 5개로 스케일해줘", "node-2에서 nginx를 node-1로 옮겨줘" 같은 명령을 즉시 실행합니다. LLM 의존 없이 로컬에서 동작하므로 네트워크 지연이나 API 비용이 발생하지 않습니다.

### 2. 멀티노드 클러스터 관리 (Master-Worker)
Master 노드가 전체 클러스터를 제어하고, Worker 노드의 Agent가 10초 간격 Heartbeat로 상태를 보고합니다. 대시보드에서 서버 IP를 입력하는 것만으로 새 노드를 추가할 수 있습니다.

### 3. 3가지 배치 전략 (Spread / Least-loaded / Binpack)
- **Spread**: 컨테이너를 노드 전체에 균등 분배 → 고가용성
- **Least-loaded**: CPU/메모리 여유가 가장 많은 노드 우선 → 성능 최적화
- **Binpack**: 기존 노드를 최대한 채운 후 새 노드 사용 → 자원 효율

### 4. 실시간 웹 대시보드 (SSE)
Server-Sent Events로 폴링 없이 실시간으로 클러스터 상태, 컨테이너 목록, 리소스 사용량이 업데이트됩니다. 별도 WebSocket 설정 없이 HTTP만으로 동작합니다.

### 5. 드래그앤드롭 컨테이너 마이그레이션
배치 현황 그리드에서 컨테이너 칩을 다른 노드로 드래그하면 자동으로 마이그레이션이 실행됩니다. 진행 상황은 프로그레스 바로 실시간 표시됩니다.

### 6. AI 채팅 어시스턴트 (대시보드 내장)
대시보드 우측 하단의 채팅 위젯에서 자연어로 인프라를 제어합니다. 명령 실행 결과를 즉시 채팅으로 확인할 수 있어 CLI에 익숙하지 않은 운영자도 쉽게 사용 가능합니다.

### 7. 자동 복구 (Auto-healing)
Worker 노드가 Heartbeat를 3회 이상 놓치면 해당 노드를 `offline`으로 전환하고, 그 노드에서 실행 중이던 컨테이너를 다른 건강한 노드로 자동 재배포합니다.

### 8. 블록체인 네트워크 원클릭 배포 (Goloop)
파라메타의 Goloop 블록체인 노드를 원클릭으로 배포합니다. 기존에 수시간이 걸리던 멀티노드 블록체인 네트워크 구성을 1분 이내로 완료합니다.

### 9. 노드 자동 프로비저닝 (SSH)
대시보드에서 서버 IP, 사용자명, SSH 키를 입력하면 원격 서버에 Docker와 Agent를 자동 설치하고 클러스터에 등록합니다. 수동 SSH 접속 불필요.

### 10. 서비스 디스커버리 & Traefik 라우팅
모든 서비스가 `<서비스명>.master-ip` 도메인으로 자동 접근 가능합니다. Traefik이 리버스 프록시와 로드밸런싱을 자동 처리합니다.

### 11. 알림 엔진 (CPU/MEM/Disk/노드 상태)
임계치 기반 자동 알림 시스템. CPU 80% 초과, 메모리 부족, 디스크 90% 초과, 노드 오프라인 등의 이벤트를 실시간으로 감지하고 대시보드에 알림을 표시합니다.

### 12. 리소스 모니터링 (컨테이너별 CPU/MEM/NET)
노드 수준뿐 아니라 개별 컨테이너의 CPU 사용률, 메모리 사용량, 네트워크 I/O를 실시간으로 모니터링합니다. 대시보드의 배치 그리드에서 각 컨테이너의 자원 사용량을 한눈에 확인할 수 있습니다.

## 기술 스택

| 카테고리 | 기술 |
|----------|------|
| **백엔드 (Go)** | Go 1.25, Chi router, Docker SDK, SQLite (WAL mode), gopsutil |
| **백엔드 (Python)** | Python 3.9, FastAPI, Docker SDK for Python, SQLite |
| **프론트엔드** | Vanilla JS, HTML5 Drag & Drop API, Server-Sent Events |
| **인프라** | Docker, Traefik (ingress/LB), dnsmasq (DNS), Alpine Linux |
| **빌드** | 단일 바이너리 (~17MB), Docker 멀티스테이지 빌드 |
| **데이터베이스** | SQLite WAL mode (동시성 지원, 별도 DB 서버 불필요) |

### 코드 규모
- **Go 구현체**: ~8,600 LoC (12개 패키지: server, runtime, scheduler, nlengine, migrate, agent, alerts, monitoring, clusterstate, discovery, models, state)
- **Python 구현체**: ~4,600 LoC (14개 모듈)
- **대시보드**: ~2,300 LoC (단일 HTML, 외부 의존성 없음)
- **총합**: ~15,500+ LoC

## 아키텍처

```
Client (Browser / CLI / AI 채팅)
  ↓ REST API + SSE (Server-Sent Events)
┌─────────────────────────────────────────────┐
│              Master Node                     │
│  ┌─────────────────────────────────────────┐ │
│  │ HTTP Server (Chi router, 100+ endpoints)│ │
│  ├─────────────────────────────────────────┤ │
│  │ NL Engine (한국어/영어 패턴 매칭)         │ │
│  │ Scheduler (spread/least-loaded/binpack)  │ │
│  │ Migration Controller (live migration)    │ │
│  │ Alert Engine (CPU/MEM/Disk/노드 감시)    │ │
│  │ Auto-Heal Engine (장애 자동 복구)         │ │
│  │ Service Discovery (Traefik 연동)         │ │
│  ├─────────────────────────────────────────┤ │
│  │ SQLite (WAL mode) — 클러스터 상태 저장    │ │
│  └─────────────────────────────────────────┘ │
└──────────────┬──────────────────────────────┘
               │ Heartbeat (10초 간격)
    ┌──────────┼──────────┐
    ▼          ▼          ▼
┌────────┐ ┌────────┐ ┌────────┐
│Worker 1│ │Worker 2│ │Worker N│
│        │ │        │ │        │
│ Agent  │ │ Agent  │ │ Agent  │
│ Docker │ │ Docker │ │ Docker │
│Traefik │ │Traefik │ │Traefik │
└────────┘ └────────┘ └────────┘
```

### 데이터 흐름
1. 사용자가 자연어 명령 입력 (예: "nginx 3개 배포해줘")
2. NL Engine이 의도(intent)와 파라미터 추출
3. Scheduler가 배치 전략에 따라 최적 노드 선택
4. Master가 해당 Worker에 REST API로 컨테이너 생성 명령 전달
5. Worker Agent가 Docker SDK로 컨테이너 생성 및 상태 보고
6. SSE로 대시보드에 실시간 반영

## 효용성 (Utility)

### 파라메타 실무에 직접 적용 가능한 솔루션

**1. 블록체인 노드 운영의 핵심 과제 해결**
파라메타는 Goloop 기반 블록체인 네트워크를 운영합니다. 새로운 테스트넷 구성, 노드 추가, 장애 대응 시 기존에는 각 서버에 SSH로 접속하여 수동으로 Docker 명령을 실행해야 했습니다. AI Container Orchestrator를 사용하면:
- "goloop 노드 4개로 블록체인 네트워크 배포해줘" → 1분 이내 완료
- 노드 장애 시 자동으로 다른 서버에 재배포 → 다운타임 최소화
- 대시보드에서 전체 네트워크 상태를 한눈에 확인

**2. Kubernetes 도입 없이 컨테이너 오케스트레이션**
- Kubernetes는 최소 3대의 Master 노드가 필요하고, etcd, API Server, kubelet 등 수십 개의 컴포넌트를 관리해야 합니다
- AI Container Orchestrator는 단일 바이너리(~17MB)로 동일한 핵심 기능을 제공합니다
- 학습 비용: Kubernetes YAML 수백 줄 vs 자연어 한 줄

**3. 비개발자의 인프라 접근성 확보**
- QA 엔지니어가 테스트 환경을 직접 구성: "테스트용 nginx 2개 올려줘"
- 운영팀이 장애 상황에서 즉시 대응: "node-3의 컨테이너를 node-1로 옮겨줘"
- 관리자가 리소스 현황 파악: 대시보드에서 실시간 모니터링

**4. 정량적 효과**

| 작업 | 기존 방식 | AI Container Orchestrator |
|------|----------|--------------------------|
| 블록체인 네트워크 배포 (4노드) | 2~3시간 (수동 SSH) | **1분** |
| 컨테이너 마이그레이션 | 30분 (수동 export/import) | **10초** (드래그앤드롭) |
| 신규 서버 클러스터 추가 | 1시간 (Docker 설치, 설정) | **3분** (자동 프로비저닝) |
| 장애 감지 및 복구 | 알림 수신 후 수동 대응 (30분~) | **자동** (30초 이내) |
| 스케일링 | CLI 명령 + 로드밸런서 수정 | **"5개로 스케일해줘"** (즉시) |

## 독창성 (Originality)

### 1. 한국어 자연어 기반 인프라 제어 — 세계 최초 수준의 접근

기존 컨테이너 오케스트레이션 도구는 모두 YAML, CLI, 또는 GUI 기반입니다. AI Container Orchestrator는 **한국어 자연어**를 1급 시민(first-class citizen)으로 지원하는 최초의 오케스트레이터입니다.

```
기존: kubectl apply -f deployment.yaml + kubectl expose + kubectl scale...
본 도구: "nginx를 3개 배포하고 외부에서 접근 가능하게 해줘"
```

### 2. Kubernetes와 Docker Compose 사이의 빈 공간을 채우는 유일한 솔루션

```
Docker Compose ←————— AI Container Orchestrator ——————→ Kubernetes
(단일 호스트)     (멀티노드, 자연어, 경량)        (대규모, 복잡)
```

Docker Swarm이 이 영역을 목표했으나 이미 Deprecated 상태입니다. Nomad는 유사하지만 자연어 제어와 한국어 지원이 없습니다. 이 포지셔닝은 현재 오픈소스 생태계에서 비어 있는 영역입니다.

### 3. 드래그앤드롭 컨테이너 마이그레이션

기존 도구에서 컨테이너를 다른 노드로 이동하려면 복잡한 CLI 명령이나 API 호출이 필요합니다. 본 도구는 대시보드에서 **컨테이너 칩을 마우스로 드래그하여 다른 노드에 놓기만 하면** 자동으로 마이그레이션이 실행됩니다. 이런 수준의 직관적 UX는 컨테이너 오케스트레이션 영역에서 유례가 없습니다.

### 4. AI 채팅 인터페이스로 DevOps 민주화

DevOps는 전문 엔지니어의 영역이었습니다. AI 채팅 인터페이스를 통해 누구나 인프라를 관리할 수 있게 함으로써, "DevOps 민주화"를 실현합니다. 이는 파라메타 내부에서 개발팀과 운영팀 간의 협업 장벽을 허물 수 있습니다.

### 5. Go + Python 듀얼 구현

동일한 기능을 Go와 Python 두 가지 언어로 구현하여, 성능이 중요한 프로덕션 환경에서는 Go 바이너리를, 빠른 프로토타이핑이나 커스터마이징이 필요한 경우에는 Python 구현체를 선택할 수 있습니다.

## 완성도 (Completeness)

### 실제 작동하는 프로덕션 수준의 구현체

**코드 품질**
- Go 구현체: 12개 패키지, ~8,600 LoC — 타입 안전, 에러 처리, 테스트 코드 포함
- Python 구현체: 14개 모듈, ~4,600 LoC — 완전한 기능 대응
- 대시보드: ~2,300 LoC — 외부 라이브러리 의존 없이 순수 HTML/JS로 구현
- 총 ~15,500+ LoC

**API 완성도**
- 100+ REST API 엔드포인트
- 컨테이너 CRUD, 클러스터 관리, 마이그레이션, 스케줄링, 모니터링, 알림, 자연어 명령 등 모든 기능에 대한 API 제공
- SSE 기반 실시간 이벤트 스트림

**테스트**
- 단위 테스트: NL 엔진, 스케줄러, 상태 관리 모듈
- API 테스트: 주요 엔드포인트 통합 테스트
- E2E 테스트 스크립트

**배포 & 운영**
- `Dockerfile` + `Dockerfile.go`: 멀티스테이지 빌드로 최적화된 이미지
- `docker-compose.yml`: Master/Worker 구성 파일 제공
- 자동 설치 스크립트 (`scripts/install-agent.sh`)
- 포괄적인 문서: README, 아키텍처 다이어그램, 배포 가이드

**안정성**
- SQLite WAL mode로 동시 읽기/쓰기 지원
- Heartbeat 기반 노드 상태 감시 (10초 간격)
- 자동 복구(Auto-healing)로 장애 시 컨테이너 자동 재배포
- 알림 엔진으로 이상 상황 사전 감지
- 에러 처리 및 구조화된 로깅 전 모듈 적용

**UX 완성도**
- 실시간 대시보드: SSE 기반, 폴링 없음
- 드래그앤드롭 마이그레이션: 직관적 조작
- AI 채팅 위젯: 대시보드 내장, 즉시 사용 가능
- 반응형 디자인: 모바일에서도 접근 가능
- 컨테이너별 리소스 사용량 시각화 (CPU/MEM/NET 바)

### 1. AI 도구 활용 전략

**사용 도구**: Claude Code (Anthropic)

**개발 프로세스**:
```
요구사항 정의 → 아키텍처 설계 → 모듈별 구현 → 통합 → 테스트 → 최적화
     ↑                                                          │
     └──────────── Claude Code와 반복적 대화 ──────────────────┘
```

### 2. 프롬프트 전략 — 효과적인 AI 협업 방법

**단계별 접근 (Top-down decomposition)**:
- 1단계: 전체 아키텍처를 자연어로 설명하고 Claude Code에게 설계 검토 요청
- 2단계: 각 모듈(NL 엔진, 스케줄러, 런타임 등)을 독립적으로 구현 요청
- 3단계: 모듈 간 통합 시 기존 코드를 컨텍스트로 제공하며 연결 코드 생성
- 4단계: 테스트 코드 생성 및 버그 수정을 AI에게 위임

**컨텍스트 관리의 핵심**:
- 매 대화마다 관련 파일과 아키텍처 결정 사항을 명시적으로 제공
- "이전에 만든 scheduler.go의 인터페이스를 유지하면서 migration 모듈을 추가해줘" 같은 방식으로 일관성 유지
- 에러 메시지를 그대로 전달하여 AI가 정확한 수정안 제시

**Go + Python 듀얼 구현 전략**:
- Python으로 먼저 프로토타입 구현 → 동작 검증
- 검증된 Python 코드를 컨텍스트로 제공하며 Go로 재구현 요청
- AI가 두 언어 간 패러다임 차이를 자동으로 처리 (async/await → goroutine, dict → struct 등)

### 3. 생산성 향상 — 정량적 성과

| 항목 | AI 미사용 예상 | Claude Code 활용 | 향상률 |
|------|---------------|------------------|--------|
| Go 백엔드 구현 (8,600 LoC) | 3~4주 | 3~4일 | **~7배** |
| Python 백엔드 구현 (4,600 LoC) | 2~3주 | 2~3일 | **~7배** |
| 대시보드 구현 (2,300 LoC) | 1~2주 | 1~2일 | **~7배** |
| Docker SDK 통합 | 1주 (문서 학습 포함) | 수시간 | **~10배** |
| 버그 수정 및 최적화 | 지속적 | 에러 메시지 전달 → 즉시 수정 | **~5배** |
| 문서 작성 | 2~3일 | 수시간 | **~5배** |

**총 개발 기간**: AI 미사용 시 약 2~3개월 → Claude Code 활용 시 **약 2주**

### 4. AI 활용에서 얻은 교훈

**효과적이었던 점**:
- **큰 기능을 작은 단위로 분해**: "마이그레이션 기능 만들어줘" 대신 "컨테이너 export 함수 → import 함수 → 상태 전환 로직 → API 엔드포인트" 순서로 요청
- **기존 코드를 컨텍스트로 제공**: 새 모듈 추가 시 관련 인터페이스/타입 정의를 함께 제공하면 일관된 코드 생성
- **에러 기반 반복**: 컴파일 에러나 런타임 에러를 AI에게 전달하면 정확한 수정안을 빠르게 받을 수 있음
- **코드 리뷰 위임**: "이 코드에서 잠재적인 동시성 이슈를 찾아줘" 같은 리뷰 요청이 효과적

**주의해야 할 점**:
- AI 생성 코드도 반드시 실행 테스트 필요 — 컴파일은 되지만 런타임 에러가 있는 경우 존재
- 복잡한 상태 관리 로직은 AI에게 전체 흐름을 명확히 설명해야 정확한 코드 생성
- 프로젝트 규모가 커지면 컨텍스트 윈도우 관리가 중요 — 관련 파일만 선별적으로 제공
